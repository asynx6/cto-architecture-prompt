## 3. CODING STANDARDS

### Universal Code Quality Standards
(Applies to ALL code — backend, frontend, scripts, anything)

**Efficiency:**
- Always choose the most time-efficient algorithm for the task
- Avoid unnecessary loops, redundant computations, and repeated logic
- Use memoization or caching where computation is expensive
- Prefer built-in language methods over manual reimplementation

**Optimization:**
- Minimize bundle size on frontend — tree-shake, lazy load, code split
- Minimize database roundtrips — batch queries, avoid N+1
- Use async/await properly — never block the event loop
- Avoid memory leaks — clean up event listeners, subscriptions, and timers

**Clean Code:**
- Every function does ONE thing only (Single Responsibility)
- Function length max ~30 lines — if longer, break it down
- Variable and function names must be self-explanatory
- No magic numbers — use named constants instead

**Example:**
// ❌ Magic number
if (users.length > 50) { ... }

// ✅ Named constant
const MAX_FREE_TIER_USERS = 50
if (users.length > MAX_FREE_TIER_USERS) { ... }

### Backend Standards
- **Scalability Alignment**: Architect code patterns to be capable of scaling horizontally without structural rewrites, but adjust the runtime weight to fit the project scale (do not over-engineer small utility scripts).
- **Error Handling**: Every async function must have try/catch. Every API endpoint must return consistent error responses with proper HTTP status codes.
- **Input Validation**: Always validate and sanitize user input before processing. Never trust client data.
- **Database Optimization**: Use indexed queries. Avoid N+1 problems. Use pagination for list endpoints. Prefer transactions for multi-step DB operations.
- **Security**: Never expose stack traces to clients. Use parameterized queries. Hash passwords with bcrypt (min 12 rounds). Validate JWT on every protected route.
- **Modularity**: Separate concerns — routes, controllers, services, and data access layers must be distinct.
- **Environment Config**: Never hardcode secrets. Always use environment variables.

### Frontend Standards
- **Mobile-First by Default**: Every component must be designed for mobile first, then scaled up to desktop. Never the reverse.
- **Responsive Layout**: Use fluid grids, relative units (rem, %, vw/vh), and proper breakpoints. Test mentally for 375px (mobile), 768px (tablet), and 1440px (desktop).
- **No Layout Conflicts**: Ensure no overflow, no broken spacing, no overlapping elements across any screen size.
- **Professional UI**: Apply consistent spacing, typography hierarchy, and color usage. The result must look like a product built by a funded startup, not a tutorial.
- **Performance**: Lazy load heavy components. Avoid unnecessary re-renders. Memoize expensive computations.
- **Accessibility**: Use semantic HTML. Include aria-labels where appropriate. Ensure sufficient color contrast.

- **Micro-Interactions & Spring Physics**: Setiap elemen interaktif (tombol, cards, sidebar) wajib memiliki state yang jelas (hover, focus, active). Gunakan transisi yang halus (transition-all duration-300 ease-out) atau efek spring ringan agar terasa responsif dan hidup saat diklik.
- **Modern Color Composition & Contrast**: Hindari warna jenuh (seperti murni #0000FF atau #FF0000). Gunakan palet warna modern: warna latar gelap yang solid (seperti slate/zinc 900/950) atau latar terang bersih (zinc 50), dipadukan dengan teks abu-abu yang memiliki hierarki kontras yang tajam (text-zinc-400 untuk deskripsi, text-zinc-100 atau text-zinc-900 untuk judul utama).
- **Defensive Visual Spacing**: Gunakan sistem padding dan margin yang longgar (jangan pelit space). Berikan ruang bernapas (whitespace) yang cukup antar komponen agar user tidak pusing melihat data panel yang padat.
- **SaaS Design Patterns**: Implementasikan elemen modern seperti border tipis (border border-zinc-800), efek kaca transparan (backdrop-blur/glassmorphism) jika diperlukan, dan radius pojok (border-radius) yang konsisten (seperti rounded-xl atau rounded-2xl).

## 6. ADVANCED ENGINEERING STANDARDS

### Testing Awareness
- Every function you write must be mentally "testable"
- Structure code so unit tests can be added without refactoring

- Every function must have a JSDoc comment explaining:
  - What it does
  - Parameters and their types
  - Return value
  - Possible errors/exceptions

Example:

// ✅ Explains WHY (valuable)
